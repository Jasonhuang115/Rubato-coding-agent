# Plan: 自进化 RAG 记忆系统

**Status:** draft | **Progress:** 0/7

## Goal

将当前静态 RAG（检索权重固定、合并策略固定）升级为自进化 RAG——记忆系统根据每次检索的实际效果自动调整权重、合并策略和事实版本，越用越准。

参考架构：RecMem（懒惰合并）+ EvoRAG（反馈反向传播）+ MemStrata（事实过期管理）

---

## Tasks

- [ ] 反馈信号采集 — 检出哪些记忆真的被用了
- [ ] 检索策略自适应 — FTS5/向量/图谱 权重自动调
- [ ] 记忆懒惰巩固 — Recurrence-based 合并替代每次会话合并
- [ ] 事实时效管理 — 同 subject+relation 出现新 object 时自动过期旧版
- [ ] 查询改写反馈学习 — 检索效果差的 query 改写策略自动调整
- [ ] 遗忘机制增强 — 结合反馈和时效双重决定遗忘
- [ ] 启动时自检报告 — 展示系统健康度

## 详细设计

### 1. 反馈信号采集

**问题**：现在不知道哪条记忆有用、哪条没用。

**方案**：在 `feedback_log` 表加字段，采集三类信号：

| 信号 | 含义 | 采集方式 |
|------|------|---------|
| `injected` | 记忆被注入 system prompt | 已有 |
| `referenced` | AI 在回复中引用了该记忆 | 会话结束后 LLM 回溯对话，判断是否引用了注入的记忆 |
| `ignored` | 注入但对话完全没提 | 注入但未被 reference 的记忆 |

**实现**：
- 扩展现有 `feedback_log` 表，加 `signal_type` 和 `metadata` 字段
- 会话结束后用廉价模型做一次回溯：给注入的记忆摘要 + 对话摘要，问 LLM "这些记忆被引用了吗？"

---

### 2. 检索策略自适应

**问题**：现在 `strategy_weights` 表的 fts5/vector/graph 权重是静态默认值。

**方案**：每次检索后根据"被引用的记忆来自哪条策略"自动调整。

```
如果 vector 路径找出的记忆被引用次数多 → vector 权重 ↑
如果 FTS5 路径找出的记忆一直没人引用 → FTS5 权重 ↓
```

**实现**：
- 在 `feedback_log` 中记录每条注入记忆来自哪个检索策略
- 每次归档时计算各策略的命中/注入比
- 每 10 次会话或每周更新一次 `strategy_weights` 表

---

### 3. 记忆懒惰巩固（RecMem 模式）

**问题**：现在 `consolidator.ts` 每次会话结束都跑——大部分时间没必要，浪费 token。

**方案**：改为触发式合并。

```
Session 1: 记忆 "碰撞检测" → count=1
Session 2: 记忆 "碰撞检测 bug" → count=2 (相似)
Session 3: 记忆 "碰撞修复"   → count=3 (触发合并)

→ LLM 将 3 条相似记忆合并为 1 条抽象原则：
  "碰撞检测需要检查物体两侧边界，单边判断会漏接"
```

**实现**：
- 新增 `pending_consolidation` 表，记录等待合并的相似记忆组
- 每次新实体写入时，用 embedding 查最相似的前 N 条
- cosine > 0.7 的归入同一组，组内计数达到阈值（默认 3）触发 LLM 合并
- 合并后新实体标记 `abstracted_from: [id1, id2, id3]`，旧实体权重降低

---

### 4. 事实时效管理（MemStrata 模式）

**问题**：配置变了、技术栈变了，旧知识还在库里占位。

**方案**：`(name, type)` 相同的实体自动建立时序链条。

```
实体: "项目端口" type=config content="8000" created_at=T1
实体: "项目端口" type=config content="8080" created_at=T2  ← 新事实

→ 自动逻辑：
  旧实体.status = "superseded"
  旧实体.superseded_by = 新实体.id
  检索时默认过滤 status=superseded 的实体
  /memory search 加 --all 可查历史版本
```

**实现**：
- `entities` 表加 `status`（active/superseded）和 `superseded_by` 字段
- `upsertEntity` 中检测同 name+type → 自动标记旧版过期
- Fusion 检索默认 `WHERE status = 'active'`

---

### 5. 查询改写反馈学习

**问题**：`rewriter.ts` 的查询改写是规则式的，不学习。

**方案**：`query_rewrite_rules` 表已有，改成基于反馈自动生成规则。

```
用户 query: "碰撞检测怎么修"
→ 改写: ["碰撞检测", "碰撞 bug", "碰撞"]
→ 检索结果中引用率最高的来自 "碰撞 bug"
→ 自动学习: query 含 "怎么修" → 加扩 "bug" 变体
```

**实现**：
- 每次检索后对比：原始 query vs 各改写变体，哪个召回被引用的记忆多
- 模式提取：当 X 类 query 的 Y 类改写效果好，记录规则
- 下次同类 query 自动生成 Y 类改写

---

### 6. 遗忘机制增强

**问题**：现在只按时间衰减，不管内容价值。

**方案**：双重决定遗忘。

```
保留分数 = 0.4 × (1 - days/60)           ← 时间衰减
         + 0.3 × feedback_score           ← 被引用次数
         + 0.2 × 是否 protected           ← 手动保护
         + 0.1 × relation_count           ← 关系丰富度
```

低于阈值且未保护的 → 标记 `deprecated`，7 天后真删除。

**方案**：rubato 启动时打印一行记忆健康度。

```
🧠 记忆健康: 实体 42 | 活跃 38 | 过期 2 | 待合并 3 | 向量搜索就绪 ✅
```

## 实施优先级

| 顺序 | 任务 | 理由 |
|------|------|------|
| **P0** | 事实时效管理 | 最简单，单项改动，立即见效 |

| **P1** | 检索策略自适应 | 依赖 P0 的反馈数据 |
| **P1** | 记忆懒惰巩固 | 省 token，且依赖 P0 信号判断合并时机 |
| **P2** | 遗忘机制增强 | 依赖反馈数据积累 |
| **P2** | 查询改写反馈学习 | 需要较多反馈数据才有效 |
| **P3** | 启动时自检报告 | 锦上添花 |
